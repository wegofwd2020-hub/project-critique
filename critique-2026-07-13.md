## Newly discovered and fixed (bugs closed in this window)

All eight items below were absent from the v1.7 critique because they were not yet known. They surfaced during live-demo QA on 2026-07-11 and were fixed in PRs #506–#508 merged 2026-07-12.

### ❌→✅ Client-side quiz grading (CRITICAL, now closed) — `8a4d3df` / PR #506

The quiz payload sent `correct_option` for every question; `POST /progress/session/{id}/answer` accepted `correct: bool`; `POST /progress/session/{id}/end` accepted `score: int`. A student could award themselves a perfect score from the devtools console. The backend never compared anything against the content store.

**Fix:** Grading is now server-side. `get_quiz_answer_key` resolves from the content store; `_strip_answer_key` removes `correct_option`/`explanation` before serving the quiz; the score is tallied in Redis (`INCR`, fire-and-forget) and read at end-of-session. `RecordAnswerRequest` no longer accepts `correct`/`correct_answer`; `EndSessionRequest` no longer accepts `score`. BREAKING CHANGE (see new risk below re Kivy).

Two subtle invariants preserved: (a) `question_id` is `q1…qN` across all three quiz sets with *different* answers per set, so the graded set is pinned per session in Redis (`quizset:{session_id}`) and persisted to `progress_sessions.quiz_set` (migration 0061); (b) `end_session` tallies from Redis not from `progress_answers` rows, which may not be flushed yet (fire-and-forget Celery).

Also closed: a `QuizPlayer` double-count bug that sent `score=6` for a 5-question quiz (the final question was included twice before `handleNext` incremented again); and the root cause of a reported 4/5 score displaying as 3/5 (the overflow was clamped silently at the top of the range, visible only in the middle).

### ❌→✅ Dev-placeholder content served to students — `82bf15e` / PR #506

The dev/demo seeders backfill missing units with stub lessons/quizzes tagged `model="dev-placeholder"`. They only write where a file is absent, so stubs persisted indefinitely and were served as real content. A tester sat a Grade 8 quiz that was entirely fabricated.

**Fix:** `get_content_file` now checks `meta.json` before serving; if `model` is `dev-placeholder` the call raises `FileNotFoundError`, producing an honest 404. The check also runs before the cache write so a placeholder never enters L2. A previous bare `except Exception` on the cache-read path is narrowed to `except (ValueError, TypeError)` — the old handler would have swallowed the new `FileNotFoundError` and continued serving stale cache.

### ❌→✅ Demo seeder pointed at a non-existent content namespace — `906d58e` / PR #506

The demo seeder created `demo-2026-g{8,9,10}` curricula and pointed classrooms at them. The pipeline writes only to `default-2026-g*`. Unit trees rendered (seeder writes `curriculum_units`) but every lesson/quiz 404'd (`content_subject_versions` and the content store had nothing under `demo-2026-*`). The seeder also wrote display names (`"Engineering"`) into `curriculum_units.subject` where the real curricula store codes (`"G10-ENG"`).

**Fix:** Demo now consumes the real platform curricula (IDs listed explicitly — grades 5–9 carry a `-stem` suffix, 10–12 do not). Re-seeding evicts any stale package first; teardown never touches `default-*` IDs.

Also adds `backend/scripts/backfill_content_registry.py` — reconciles `curricula` / `curriculum_units` / `content_subject_versions` to what is actually on disk for any curriculum whose pipeline writes succeeded but DB writes were silently skipped (pitfalls #28 RLS bypass, #29 `datetime`-vs-string, #30 `unit_name NOT NULL`). 151 units across 11 curricula were on disk and invisible to the application. Skips placeholder units. Idempotent; dry-run by default.

### ❌→✅ Content backfill subject-key mismatch — `66d1b0e` / PR #508

`backfill_content_registry.py` initially keyed `content_subject_versions.subject` from the data-JSON subject code (`"G10-MATH"`). But `content/service.py::check_content_published` matches that column against `curriculum_units.subject`, which for some seeds holds a display name (`"Mathematics"`). When they diverged, the publish row never matched and the content 404'd even after a "successful" backfill. Found live: `default-2026-g10` had 19 real units on disk; after backfill it had four published rows keyed by code while its units were keyed by display name. Every Grade 10 lesson and quiz still 404'd.

**Fix:** The publish key is now read back from `curriculum_units` for on-disk units, matching whatever the units actually store. The data-JSON is used only to resolve the human-readable `subject_name`.

### ❌→✅ Analytics Celery poison-task (retry loop, now closed) — `e1f7b00` / PR #506

`end_lesson_view` did `row["view_id"]` on the result of an `UPDATE … RETURNING` without checking for `None`. When the update matched no row (the lesson had 404'd so `startLessonView` never fired), `write_lesson_end_task`'s blanket `except Exception: raise self.retry(...)` retried it forever. Surfaced when the Celery worker came back after four weeks down with 29 queued lesson-end writes whose `lesson_views` rows had never been created.

**Fix:** A missing view is a permanent condition. The function now returns `updated: False` instead of raising, ending the retry loop.

### ❌→✅ Pipeline LLM error masking + transient/permanent retry conflation — `92147f6` / PR #506

A Grade 8 build failed 18 of 20 units, logging only `"anthropic call failed"` — three times per content type, 54 wasted API calls. The actual cause was an exhausted Anthropic credit balance (HTTP 400), a message destroyed by two layers of re-wrapping. The retry loop treated every 4xx as transient.

**Fix:** `_describe()` now walks `__cause__`/`__context__` so the failure message surfaces. `PermanentLLMError` (subclasses `RuntimeError`, backward-compatible) marks 400/401/403/404 — the retry loop re-raises immediately. 429 is deliberately excluded (rate limits are transient). Same root cause as the analytics poison task: `except Exception: retry` treating every failure as transient.

### ❌→✅ aiosmtplib CVE-2026-53533 (SMTP command injection) — `2cd5b12` / PR #507

`aiosmtplib 3.0.1` — `mail()`/`rcpt()`/`vrfy()`/`expn()` sent caller-supplied addresses to the server without rejecting embedded CR/LF, allowing command injection. Directly reachable: user-supplied addresses are passed into `aiosmtplib.send()` from registration, demo requests, and password reset.

**Fix:** Upgraded to `5.1.1`, which rejects C0 control characters in command arguments. Only breaking change is a Python-version floor (≥3.10; we run 3.11). Email and auth suites pass.

### ❌→✅ Alembic silently targeting the test database — `50091474` / PR #506

`docker compose exec api alembic upgrade head` migrated `studybuddy_test`, not the dev DB, reported success, and left the dev DB untouched (pitfall #34 — a nastier variant of pitfall #18 because it strikes *after* you ran the migration).

**Fix:** `alembic/env.py` now prints the target database on every run and warns when `TEST_DB_URL` is the reason. CLAUDE.md updated with `-e TEST_DB_URL=` flag in all manual-migration examples.

## Newly introduced risks (opened in this window)

### ⚠️ NEW — Kivy mobile client broken for quiz functionality (BREAKING)

The server-side grading change (`8a4d3df`) removed `correct_option` and `explanation` from the quiz payload. The Kivy mobile client (`mobile/src/ui/QuizScreen`) grades locally from `correct_index` derived from `correct_option` — a field that no longer exists in the response. The client will silently receive `null`/`undefined` and produce wrong quiz behaviour (likely always shows "incorrect" or crashes the answer handler).

The commit acknowledges this: *"The Kivy mobile client still grades locally from a `correct_index` it will no longer receive — it is parked behind the Expo rewrite (Epic 3) and needs the same treatment."* The mobile app is not currently promoted to real students, but it is a regression for any operator testing the Kivy path. Add to the P2 list.

### ⚠️ NEW — Pass-rate denominator labelling gap on reporting screens

Documented in `4941476` (QA report Q4 answer): the **Overview** screen computes pass rate over first attempts only; the **At-Risk** screen uses all-attempts-lifetime. The same student's pass rate legitimately differs between the two screens with no UI label to explain the difference. A teacher reading both screens will see contradictory numbers. Low severity but will generate support questions in school pilots. Add to the P3 list.

### ⚠️ NEW — ecdsa PYSEC-2026-1325 (unfixable transitive advisory)

Transitive from `python-jose`. Upstream declines to fix timing side-channels, stating the library is not intended for high-security use. No ECDSA operation is performed here (HS256 + RS256 only). Added to the `pip-audit` ignore list with documented justification in `2cd5b12`. **Revisit if an EC algorithm is introduced or `python-jose` is replaced.** The advisory is real but the exposure is zero given the current algorithm choices.

### ℹ️ OPERATIONAL — Live demo now public (`demo.usestudybuddy.com`)

The demo is now linked from the README and the domain is live. This changes the threat surface: public-facing endpoints are now reachable by anyone, not just local testers. The existing rate-limit gaps (slowapi in-process coexistence, `APP_ENV` unasserted) are now live risks, not just theoretical ones. The two P1 items covering these should be treated as more urgent.
