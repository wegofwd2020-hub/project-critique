## ‚úÖ Items resolved since v1.7

### 1. Self-serve password reset for local (school-provisioned) users ‚Äî #444

`POST /auth/forgot-password` now branches on account type: for local users it mints a single-use Redis token (`local_pw_reset:{token}`, 1 hr TTL) and sends a reset-link email; Auth0 users follow the existing hosted flow. New `POST /auth/reset-password` consumes the token, sets `password_hash`, clears `first_login`, and burns the token. Always returns 200 (no email enumeration). Pitfall #23 (`app.current_school_id='bypass'`) applied correctly on both unauthenticated DB lookups.

`backend/tests/test_auth_local_reset.py` covers: issue ‚Üí reset ‚Üí login (old password rejected, `first_login` cleared), invalid token, single-use enforcement, short-password validation, Auth0 fallback path.

Previously there was no self-serve recovery path for school-provisioned users; they had to wait on a school admin.

## üÜï New issues since v1.7

### N1 ‚Äî Demo paywall exposed in live walkthrough (2026-06-27)

During a demo session, student accounts returned HTTP 402 on every lesson (`_FREE_TIER_LESSON_LIMIT = 2` cap, no active school subscription on the demo school). Quizzes and tutorials were unaffected (not gated). Fixed by activating placeholder subscriptions (`school_subscriptions` rows with `status='active'`) on both demo schools. Documented in `docs/DEMO_FIX_2026-06-27_lesson_paywall.md`.

Two residual issues:
- **#504 (open):** When a free-tier student hits the lesson cap, the UI shows a generic error instead of an explicit "you've reached your free lesson limit ‚Äî upgrade" message. The paywall is real business logic; the UX is confusing. Track to close before first paying-school onboarding.
- **Demo runbook gap:** The demo paywall was invisible until a live walkthrough hit it. The demo setup doc should include a verification step confirming `school_subscriptions.status = 'active'` before handing off any demo to a prospect.

### N2 ‚Äî 4 Starlette CVEs formally ignored (not fixed) ‚Äî #379

As part of the CI-unblock work (#499), 4 starlette CVEs (CVE-2026-48817, CVE-2026-48818, CVE-2026-54282, CVE-2026-54283) were added to the `pip-audit` ignore list alongside the pre-existing PYSEC-2026-161. The rationale is sound (fixes require starlette 1.x, incompatible with `fastapi 0.123` and `prometheus-fastapi-instrumentator 7.1.0`; the specific vectors don't apply given JWT Bearer auth and limited `url.path` usage). However, the ignore list is now 5 entries deep on starlette alone. Track #379 for a resolution path ‚Äî specifically whether upgrading `prometheus-fastapi-instrumentator` to a starlette-1.x-compatible release would unblock the starlette upgrade.

### N3 ‚Äî `attempt_number` historical data inconsistency (legacy of #465 fix)

The #465 fix correctly changes `attempt_number` to be assigned at session **completion** rather than session **open**. However, sessions created before this fix were numbered at open-time. Any historical data where a unit has multiple partially-overlapping sessions may have incorrect `attempt_number` values in `progress_sessions`. This is low-impact now (demo data, no paying schools), but should be documented and addressed (via backfill or clear documentation of the semantic change) before teacher reporting features expose historical attempt-number data.

### N4 ‚Äî Keepalive fetch for lesson duration persistence ‚Äî browser-dependent (#464)

The fix for lesson duration persistence (`endLessonViewBeacon()`) uses `fetch` with `keepalive: true` instead of `sendBeacon` (to allow the `Authorization` header). The keepalive fetch has a 64 KB body-size limit per the Fetch spec. If the lesson-end payload (including the Authorization JWT) approaches this limit ‚Äî possible in environments with large JWTs or extended payloads ‚Äî the keepalive call will silently fail and `duration_s` will be NULL. Monitor under real usage; add a server-side NULL-to-zero default in reporting if durations remain unreliable.

Additionally, `endLessonViewBeacon()` fires on both `unmount` and `pagehide` ‚Äî if both fire in sequence (navigation away), the endpoint may receive two calls for the same `lesson_view_id`. Confirm the endpoint is idempotent on duplicate `lesson_view_id` writes (or debounce the beacon).

### N5 ‚Äî PROGRESS.md / EPIC_16 status drift

`docs/PROGRESS.md` (auto-generated nightly) reports Epic 16 as "üîú Ready to build ‚Äî start 2026-05-03". `CLAUDE.md` reports "Epic 16 S-1 through S-5 shipped." The `EPIC_16_public_site_redesign.md` file fed to the generator appears to still say "Ready to build". This is documentation drift ‚Äî the auto-generator is reading stale epic status. The nightly script would need the epic file updated with `Status: ‚úÖ Complete` to reflect reality. Low severity but creates false impressions for anyone reading the progress dashboard.

### N6 ‚Äî Locale JWT re-mint creates potential cross-tab / mobile inconsistency (#470)

`PATCH /auth/settings` now returns a new token when locale changes. The web client stores it as `sb_token` and the next request uses the new locale. However:

- **Multiple tabs:** tabs open before the settings save still hold the old JWT. They will continue to request content in the old locale until the old token expires or the tab is refreshed. No notification mechanism exists.
- **Mobile client:** the Kivy mobile app's `SyncManager` and `LocalCache` use JWTs stored on disk. If a student changes their locale on the web, the mobile app will not receive the re-minted JWT and will continue requesting content in the old locale until next login. This is the "cross-client auth continuity" gap flagged as P2 in v1.7, now with a concrete failure mode.
