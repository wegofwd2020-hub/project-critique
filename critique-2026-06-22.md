### ✅ Self-serve password reset for local (school-provisioned) users (#444, #489, #491)

Local-auth users (school-provisioned teachers and students) had no self-serve recovery path — `POST /auth/forgot-password` only reached Auth0, leaving school-provisioned users dependent on a school admin. Now:
- `POST /auth/forgot-password` branches on account type: local users get a Redis one-time token (`local_pw_reset:{token}`, 1 hr TTL) + a reset-link email; Auth0 users fall through as before. Still always returns 200 (no enumeration).
- `POST /auth/reset-password` consumes the token, sets `password_hash`, clears `first_login` (self-serve reset is a deliberate choice, unlike admin provisioning), burns the token (single-use). Returns 400 on invalid/expired.
- Correctly stamps `app.current_school_id='bypass'` for the unauthenticated DB lookup (pitfall #23 compliance).
- Frontend: `/reset-password` aligned its minimum-length rule to 12 chars to match backend (was 8, causing confusing 422s).
- Contract drift caught and fixed in follow-up #491: `POST /auth/reset-password` added to `openapi.json` + `types.gen.ts`.
- Tests: `test_auth_local_reset.py` covers token issuance, Auth0 fallback, full forgot→reset→login roundtrip, old-password rejection, `first_login` clearing, invalid token, single-use enforcement, short-password validation.

### ✅ Units without content gated with honest UX (#468, #469, #495)

Students could previously select units whose content was never generated, then see a generic red "Could not load lesson. Please try again." (wrongly implying a transient error). Fixed:
- `GET /curriculum/tree` now returns a per-unit `has_content` boolean, probed against the content store for `lesson_en.json`. School fork units skip the probe (their content is served from DB overrides).
- Web: Subjects page greys out unavailable units with a "Coming soon" badge and prevents the click. Curriculum Map dims the spine and shows "not published yet" instead of action buttons.
- Content pages (`/lesson`, `/quiz`, `/tutorial`, `/experiment`) now use `contentErrorMessage()` (`web/lib/content-error.ts`, 31 LOC) to distinguish a 404 "not yet available" from a transient failure — both rendered in age-appropriate non-technical copy (Content Rule #5 compliance).
- Test: `test_curriculum_content_flag.py` (54 LOC) covers present/absent units, empty input, and storage-error defaults-to-available.

### ✅ Locale change applies immediately to content (#470, #496)

Selecting a language and saving updated `students.locale` in the DB but the change never reached content until logout/login — locale is authoritative from the JWT, and the old token was still in use. Fixed:
- `PATCH /auth/settings` re-mints the student JWT with the new locale (all other claims preserved) and returns it as `{token}` only when locale changed. Non-locale saves return no token.
- Frontend `saveAccountSettings` stores the returned token as `sb_token`; the axios client reads it per-request.
- Frontend helper copy corrected: "Content is served in your selected language" → honest "where a translation exists, in English otherwise" (Spanish content is not yet generated).
- Test: `test_settings_locale.py` asserts locale change returns a decodable token with the new locale; non-locale and invalid-locale changes return none.

**Security note:** JWT re-minting from a non-auth endpoint is a new pattern. The implementation correctly preserves all existing claims and doesn't elevate privileges — worth verifying in future audits that the claim-copy logic can't be tricked by a malformed settings payload.

## New Issues Introduced Since v1.7

### ⚠️ NEW — `has_content` store probe uncached on every `/curriculum/tree` call

`GET /curriculum/tree` now probes the content store for `lesson_en.json` per unit to compute `has_content`. The commit notes school forks skip the probe, but for platform curricula this is one storage I/O per unit per tree request. For a Grade 11–12 curriculum with ~60 units, that's 60 S3 HEAD requests per cold tree call. There is no visible caching of `has_content` in the commit (no Redis key, no L1 TTLCache entry). At scale (many concurrent students on a cold cache), this could become a hot-path bottleneck and directly violates the "hot read path touches zero DB queries on cache-warm requests" rule extended to storage probes. Recommend: cache `has_content` in Redis with a TTL aligned to content pipeline publish cadence (e.g., 5 min), or derive from the `content_subject_versions` table (already in DB) rather than probing the store.

### ⚠️ NEW — Public `/quality` page claims WCAG 2.1 AA while 3 axe rules remain disabled

PR #475 ships `web/app/(public)/quality/page.tsx` (332 LOC) + `docs/COMPLIANCE_ONE_PAGER.md` (115 LOC) + `docs/SPEC_content_verified_badge.md` (289 LOC), all advertising WCAG 2.1 Level AA compliance. Issue #189 — which disables `color-contrast`, `html-has-lang`, and `document-title` axe rules across the E2E suite — is still open. If a school procurement officer runs an automated accessibility check, they may find the claim unsubstantiated. The `/quality` page raises the stakes on closing #189; this is no longer just an internal gap.

### ⚠️ NEW — `SPEC_content_verified_badge.md` is a spec for unshipped work

The 289-LOC spec describes a "Content Verified" badge to be shown on lessons/quizzes (with verification criteria, pipeline integration, and UI placement). It is a spec document only — the feature is not implemented. The spec file living in `docs/` alongside shipped ADRs could be misread as documentation of a live feature. Consider moving unshipped specs to `docs/epics/` or marking clearly as "Proposed / Not Shipped".

### ⚠️ CARRY-FORWARD ELEVATED — Starlette CVEs tracked but pending a major dependency bump

#499 correctly ignores the 4 starlette CVEs with justification, but the fix path (starlette 1.x) requires upgrading fastapi, which in turn requires bumping `prometheus-fastapi-instrumentator`. This is a three-package coordinated upgrade. Tracked as #379. The risk vectors cited in #499 ("JWT Bearer auth; `url.path` used only for metrics exclusion") are credible mitigations for now, but this debt should be scheduled before the production launch milestone — not deferred indefinitely.
