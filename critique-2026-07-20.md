## Newly shipped (this window)

### ✅ NEW — 4 nightly CI scanners recovered and live (PR #352, merged 2026-07-15)

Four workflows drafted on 2026-04-20 had been stranded on stale topic branches ~1,900 files behind `main`. Cherry-picked onto a fresh branch and merged:

| Workflow | Schedule (UTC) | What it does |
|---|---|---|
| `doc-drift.yml` | 03:00 | Scans `studybuddy-docs` for FastAPI route + env-var references no longer present in `backend/`. Requires `DOCS_REPO_TOKEN` secret; skips gracefully without it. |
| `glossary-drift.yml` | 04:45 | Flags ALLCAPS acronyms in `studybuddy-docs` prose not defined in `GLOSSARY.md`. Same `DOCS_REPO_TOKEN` gate. |
| `nightly-vuln-scan.yml` | 05:30 | `pip-audit --no-deps` on `backend/requirements.txt`. Direct pins only. |
| `nightly-secrets-scan.yml` | 06:30 | `gitleaks` against full git history. `.gitleaksignore` allowlist covers 4 reviewed test/dev fixtures. |

Each scanner upserts a single open issue (labeled `doc-drift` / `glossary-drift` / `vuln-scan` / `secrets-scan`) and auto-closes when clean.

**Two open caveats:**
1. The commit notes a built-in 2-week re-evaluation: *"if they aren't producing actionable signal, delete them"* — the sister project (thittam) ran the same scanners for 23 days with very low signal volume. Signal quality here is unknown until they accumulate history.
2. `nightly-vuln-scan.yml` uses `--no-deps` — only direct pins are audited. Transitive CVEs still rely on the full-dep `pip-audit` in the CI `test.yml`. The two scanners are **additive**, not redundant: nightly direct-only = fast signal; CI full-dep = comprehensive gate.

### ✅ NEW — Privileged seed passwords now env-overridable (PR #511, merged 2026-07-15)

Three seed scripts had hardcoded literal passwords for privileged accounts:

| Script | New env var |
|---|---|
| `backend/scripts/seed_demo_test_account.py` | `DEMO_STUDENT_PASSWORD` |
| `backend/scripts/setup_dev.py` | `DEV_ADMIN_PASSWORD` |
| `backend/scripts/seed_phase_a_dev.py` | school admin passwords from env |

Pattern matches the existing `seed_super_admin.py` convention (`os.environ.get` with a dev default). Non-privileged fixtures (teacher/student rows, milfordwaterford demo roster) are left as literals — they seed throwaway demo accounts where per-account env vars would add noise without security value. Dev defaults are unchanged; local flows are unaffected.

**Net effect:** A real deployment can now inject strong admin credentials via env without editing committed code. This is quiet but real — it closes the gap where `detect-secrets` would not catch a hardcoded default because the value was innocuous-looking in isolation.
