## 🆕 New Issues since v1.7

### N-1 — `wegofwd-llm` git+ dependency: supply-chain and deploy risk (P2)

`backend/requirements.txt` now includes:

```
wegofwd-llm[anthropic] @ git+https://github.com/wegofwd2020-hub/wegofwd-llm@v0.1.2
```

Three concerns:

1. **Docker image must have `git` installed.** This was already demonstrated as a live incident: PR #432 (incident #428) added `git` to the Dockerfile specifically because the `git+` pip dep failed during deploy. Adding `git` to a production image expands the container attack surface (git binary + its transitive deps).

2. **Extras mismatch.** `requirements.txt` declares only `[anthropic]` but `registry.py` routes all three providers (anthropic, openai, google) through `wegofwd-llm`. The adapter imports `from wegofwd_llm import build_provider` lazily, so if `[openai]` or `[google]` extras are needed for those vendor SDKs to install, they will silently fail at runtime when a school switches provider. The `requirements.txt` comment still says *"google stays native for now"* — this comment is stale; google is fully migrated per `registry.py` (`_wegofwd_adapter.py` docstring, verified live 2026-06-09).

3. **No PyPI publication.** Availability is conditional on GitHub being reachable and the `v0.1.2` tag remaining stable on the sibling repo. There is no `pip.lock` / `uv.lock` that would freeze transitive deps of `wegofwd-llm` itself.

**Recommended actions:** (a) publish `wegofwd-llm` to PyPI or a private registry so the Docker build is network-agnostic; (b) verify `[anthropic]` covers all three vendor SDKs or add the correct extras; (c) remove the stale comment; (d) add a `wegofwd-llm` provider smoke-test to CI that instantiates all three providers (with mock keys) to catch import-time errors before deploy.

### N-2 — Starlette CVE ignore list has grown from 1 to 5 entries (P2, tracked #379)

v1.7's pip-audit ignore list held one entry (PYSEC-2026-161). The `#499` CI fix added four more starlette CVEs to the ignore list: CVE-2026-48817, CVE-2026-48818, CVE-2026-54282, CVE-2026-54283. The fix (starlette 1.x) is blocked by `fastapi==0.123.0` and `prometheus-fastapi-instrumentator==7.1.0`. Issue #379 tracks this but has seen no movement. Each new ignore entry is a deliberate acknowledgment that these vectors don't apply to the current auth/routing surface — the rationale (`JWT Bearer auth`, `url.path` used only for metrics exclusion) is documented in the commit. However, the growing list signals that the fastapi/instrumentator pin is accumulating security debt. If fastapi 1.x ships a compatible release before launch, unblocking this should be prioritised.

### N-3 — Demo free-tier 402 message still non-informative (#504 open) (P2)

`#505` applied a DB-level fix (enrolling the demo school into an active subscription to clear the 2-lesson cap). Frontend ticket #504 — the error message students see when content returns HTTP 402 — remains open. Until #504 lands, a demo student who triggers any entitlement failure sees a generic red "Could not load lesson" that violates Content Rule #5 (non-technical, age-appropriate messages). Given that the demo is a primary sales channel (school admins evaluating the product), this is higher-impact than a typical UX gap.

### N-4 — PROGRESS.md epic-status stale for shipped epics (P3)

The auto-generated `docs/PROGRESS.md` shows Epic 16 ("Public Site Redesign") as `🔜 Ready to build — start 2026-05-03` even though CLAUDE.md confirms S-1–S-5 shipped. The generator attributes commits to epics by ticket prefix; epics that shipped before the script was wired in don't appear as complete. Not a correctness issue but misleading to anyone using PROGRESS.md as a status board. Additionally, task #180 in the "Agent orchestration — in flight" section still shows as `escalated` even though #501 added its test coverage (the task predated the fix and was not updated).

### N-5 — Old JWT persists after locale change (P3 / session hygiene)

`PATCH /auth/settings` (fix #470) re-mints a student JWT with the new locale and returns it. The old JWT is not revoked (no `jti` blocklist). A student with the app open in a second tab or on the mobile app will continue receiving content in the old locale until their token expires (default `JWT_ACCESS_TOKEN_EXPIRE_MINUTES`). For locale changes this is cosmetic, not a security issue. However, it is the same pattern that would be inadequate if the same code path ever carries a privilege change. Worth noting before any higher-stakes claim is put in this JWT re-mint path.
